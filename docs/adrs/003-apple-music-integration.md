# Architecture Decision Record: Apple Music Integration

**Title**: Apple Music Integration with Shared Dispatch Layer
**Status**: Proposed
**Date**: 2026-03-26
**Decision Makers**: Troy (Project Lead), Claude Code (Development Assistant)

## Context and Problem Statement

Setlistify currently supports Spotify as the only music service for playlist creation. Users who prefer Apple Music have no way to use the core feature. We want to add Apple Music with full feature parity: song search, playlist creation, and playlist display.

Adding a second provider raises several architectural questions:
- How should the web layer call music service operations without knowing which provider is active?
- How should user sessions be modeled for providers with different authentication models?
- How should session lifecycle (storage, restoration, refresh) be handled per provider?
- How should OpenTelemetry attributes be named to support cross-provider observability?

### Constraints

- No database — everything in-memory with GenServer (established by ADR-001)
- One service active at a time per user
- Must preserve full test coverage with Hammox mocking pattern
- TDD approach throughout
- Apple Music uses MusicKit JS (browser-initiated auth), not server-side OAuth redirect
- Apple Music user tokens are valid ~6 months; no server-side refresh endpoint exists
- **Apple Music provides no stable user identity primitive** — the Music User Token is opaque and app-scoped; no `/me` profile endpoint exists. A UUID is generated by the application on first auth and stored in the session cookie.

## Decision Drivers

1. **Feature parity**: Apple Music users should have an identical experience to Spotify users
2. **Minimal web layer coupling**: LiveViews should not need to know which provider is active
3. **Test compatibility**: Existing Spotify mock infrastructure should require minimal changes
4. **Observability**: Cross-provider trace filtering should be possible in Grafana/Tempo
5. **Provider isolation**: Spotify-specific session logic must not bleed into Apple Music and vice versa
6. **Incremental delivery**: Apple Music auth uncertainty should be validated before broad refactoring

## Considered Options

### Option 1: Unified `MusicService.Account` struct with `:service` field

A single struct used throughout the app with `service: :spotify | :apple_music` and all provider-specific fields (`refresh_token`, `expires_at`, `user_token`, `storefront`) present but nil when not applicable.

**Pros**:
- Single type to track across the codebase
- Simple pattern matching on `:service` field for dispatch

**Cons**:
- Nil fields for non-applicable providers create confusion about what's required
- `@enforce_keys` cannot encode per-provider required fields
- Merging semantically different fields (`access_token` vs `user_token`) obscures intent
- The existing `MusicAccount` stub was never used — there's no existing code to migrate

### Option 2: Separate structs per provider + `MusicService.API` thin dispatch

Keep `Spotify.UserSession` unchanged. Add `AppleMusic.UserSession` with Apple Music-specific fields. A `MusicService.API` module provides the shared interface the web layer calls, dispatching to `Spotify.API` or `AppleMusic.API` by pattern-matching on the session struct type.

**Pros**:
- Each struct enforces exactly the fields that provider requires
- No nil fields; struct definition is self-documenting
- Existing Spotify mock (`Spotify.API.MockClient`) requires no changes — dispatch chain preserved
- Pattern matching on struct type is idiomatic Elixir
- Provider modules remain independently testable

**Cons**:
- Two struct types to maintain
- Dispatch module adds a layer (but it's trivially thin)

### Option 3: Protocol dispatch

Define a `Setlistify.MusicService` protocol implemented for each `UserSession` struct type.

**Pros**:
- Idiomatic Elixir for open polymorphism
- Automatic dispatch on data type

**Cons**:
- Hammox requires a `@behaviour` module, not a Protocol — mocking breaks with this approach
- Protocols are designed for open extensibility; we have a fixed, known set of providers

## Decision

We will implement **Option 2: Separate structs per provider + `MusicService.API` thin dispatch**.

The `MusicService.API` module defines shared `@callback` declarations (for Dialyzer + documentation) and provides public dispatch functions. Provider API modules (`Spotify.API`, `AppleMusic.API`) add `@behaviour Setlistify.MusicService.API`, which lets Dialyzer verify they implement all shared operations. The `impl()` pattern and tracing stay inside each provider's API module, keeping the existing mock chain intact.

The unused `Setlistify.MusicAccount` stub is deleted.

## Implementation Details

### Session Structs

```elixir
# Unchanged
defmodule Setlistify.Spotify.UserSession do
  @enforce_keys [:access_token, :refresh_token, :expires_at, :user_id, :username]
  defstruct [:access_token, :refresh_token, :expires_at, :user_id, :username]
end

# New
defmodule Setlistify.AppleMusic.UserSession do
  @enforce_keys [:user_token, :user_id, :storefront]
  defstruct [:user_token, :user_id, :storefront]
end
```

`username` is absent from `AppleMusic.UserSession`. The Apple Music API has no user profile endpoint — there is no display name to fetch. Any provider-aware UI that currently renders `user_session.username` must handle the Apple Music case explicitly (see Sign-in UX below).

The `:storefront` field (e.g., `"us"`, `"gb"`) is required because Apple Music catalog endpoints are scoped by storefront: `/v1/catalog/{storefront}/search`. It is read from the MusicKit JS instance after authorization — no separate API call is needed (see Authentication Flow below).

The developer token is **not** stored in `UserSession` — it is app-global, not per-user, and is managed separately by `AppleMusic.DeveloperTokenManager` (see below).

### Apple Music User Identity

**The Apple Music API provides no stable user identifier.** There is no `/v1/me` profile endpoint equivalent to Spotify's. The Music User Token is an opaque, app-scoped, device-scoped credential. It is not a JWT, cannot be decoded to extract a subject claim, and does not encode a user ID.

**Approach**: The application generates a `UUID.uuid4()` for each Apple Music user on first authentication. This UUID is stored in the session cookie alongside the encrypted user token and storefront, and reused on every subsequent page load via session restoration.

Lifecycle:
1. **First auth** — LiveView handler generates UUID, stores in session cookie as `:user_id`
2. **Subsequent requests** — `RestoreAppleMusicToken` reads UUID from session cookie; no API call required
3. **Logout** — session cleared; UUID discarded
4. **Re-auth** — new UUID generated; prior UUID is gone (no persistence without a database)

The UUID is application-internal only. It is never sent to Apple's servers and has no meaning outside this application. It serves as a stable key for the Registry and session process during a single session.

This also means there is no cross-device or cross-browser session continuity for Apple Music users. A user who signs in on two devices will have two independent UUIDs and two independent Registry processes. This is consistent with the no-database constraint and mirrors how Spotify sessions work when a refresh token is not present.

### Shared Dispatch: `Setlistify.MusicService.API`

> **Pre-implementation task**: The `@callback` for `search_for_track` below reflects the post-Phase-1 shape (without `preview_url`). Before writing this module, confirm Phase 1's `preview_url` removal is complete and all tests pass against the new callback shape.

```elixir
defmodule Setlistify.MusicService.API do
  alias Setlistify.{Spotify, AppleMusic}

  @type user_session :: Spotify.UserSession.t() | AppleMusic.UserSession.t()

  @callback search_for_track(user_session(), String.t(), String.t()) ::
              nil | %{track_id: String.t()}

  @callback create_playlist(user_session(), String.t(), String.t()) ::
              {:ok, %{id: String.t(), external_url: String.t()}} | {:error, atom()}

  @callback add_tracks_to_playlist(user_session(), String.t(), [String.t()]) ::
              {:ok, atom()} | {:error, atom()}

  def search_for_track(%Spotify.UserSession{} = s, artist, track),
    do: Spotify.API.search_for_track(s, artist, track)
  def search_for_track(%AppleMusic.UserSession{} = s, artist, track),
    do: AppleMusic.API.search_for_track(s, artist, track)

  def create_playlist(%Spotify.UserSession{} = s, name, desc),
    do: Spotify.API.create_playlist(s, name, desc)
  def create_playlist(%AppleMusic.UserSession{} = s, name, desc),
    do: AppleMusic.API.create_playlist(s, name, desc)

  def add_tracks_to_playlist(%Spotify.UserSession{} = s, id, tracks),
    do: Spotify.API.add_tracks_to_playlist(s, id, tracks)
  def add_tracks_to_playlist(%AppleMusic.UserSession{} = s, id, tracks),
    do: AppleMusic.API.add_tracks_to_playlist(s, id, tracks)
end
```

The `@type user_session()` alias is required for Dialyzer to check the callback constraints at the dispatch layer. Without it, `user_session` in `@callback` is an unresolved type and the behaviour check provides no type safety.

`get_embed/1` is excluded from this module. The caller (`playlists/show_live.ex`) already knows the provider from the `?provider=` URL parameter, so it dispatches directly:

```elixir
def handle_params(%{"provider" => "spotify", "url" => url}, ...) ->
  Spotify.API.get_embed(url)
def handle_params(%{"provider" => "apple_music", "url" => url}, ...) ->
  AppleMusic.API.get_embed(url)
```

### Track ID Abstraction

The field is believed to be fetched but never consumed — this audit confirms it before the callback type is changed. If any read site is found, remove it as a prerequisite.

The search result type is simplified from `%{uri: String.t(), preview_url: String.t()}` to `%{track_id: String.t()}`.

`preview_url` is removed entirely — it is fetched from the Spotify API but never read anywhere in the application (no template, no LiveView, no other module references it).

`:track_id` is provider-opaque:

- Apple Music: catalog ID (`"1440857781"`)

The provider's `add_tracks_to_playlist/3` implementation knows how to interpret it. The web layer treats it as an opaque reference.

### Apple Music Developer Token Management

The developer token is an ES256-signed JWT (signed with the Apple `.p8` private key) required on every Apple Music API call. It is **app-global** — shared across all users, not per-user — so it does not belong in `UserSession`.

`Setlistify.AppleMusic.DeveloperTokenManager` is a GenServer started at application boot. It generates the token on `init`, schedules proactive self-refresh before the `exp` claim (same `@refresh_threshold` pattern as `Spotify.SessionManager`), and exposes `get_token/0` for normal use and `regenerate_token/0` for the defensive 401 path.

```elixir
defmodule Setlistify.AppleMusic.DeveloperTokenManager do
  use GenServer

  # Refresh 5 minutes before expiry, same pattern as Spotify.SessionManager
  @refresh_threshold 5 * 60
  # Default TTL — well under the 6-month Apple maximum
  @default_ttl_seconds 30 * 24 * 60 * 60

  def start_link(_opts), do: GenServer.start_link(__MODULE__, [], name: __MODULE__)

  # Returns the current cached token. Called on every Apple Music API request.
  def get_token, do: GenServer.call(__MODULE__, :get_token)

  # Forces immediate token regeneration and returns the new token. Called only
  # on a 401 response from ExternalClient. Cancels the pending scheduled refresh
  # and reschedules from the new expiry — prevents a double refresh if the
  # scheduled timer fires shortly after.
  def regenerate_token, do: GenServer.call(__MODULE__, :regenerate_token)

  @impl true
  def init(_), do: {:ok, %{token: nil, expires_at: nil, timer_ref: nil}, {:continue, :generate_token}}

  @impl true
  def handle_continue(:generate_token, state) do
    {:ok, token, expires_at} = generate_and_sign()
    timer_ref = schedule_refresh(expires_at, state.timer_ref)
    {:noreply, %{state | token: token, expires_at: expires_at, timer_ref: timer_ref}}
  end

  @impl true
  def handle_call(:get_token, _from, state), do: {:reply, state.token, state}

  @impl true
  def handle_call(:regenerate_token, _from, state) do
    {:ok, token, expires_at} = generate_and_sign()
    timer_ref = schedule_refresh(expires_at, state.timer_ref)
    new_state = %{state | token: token, expires_at: expires_at, timer_ref: timer_ref}
    {:reply, token, new_state}
  end

  @impl true
  def handle_info(:refresh_token, state) do
    {:ok, token, expires_at} = generate_and_sign()
    timer_ref = schedule_refresh(expires_at, state.timer_ref)
    {:noreply, %{state | token: token, expires_at: expires_at, timer_ref: timer_ref}}
  end

  defp generate_and_sign do
    team_id  = Application.fetch_env!(:setlistify, :apple_music_team_id)
    key_id   = Application.fetch_env!(:setlistify, :apple_music_key_id)
    pem      = Application.fetch_env!(:setlistify, :apple_music_private_key)

    now        = System.system_time(:second)
    expires_at = now + @default_ttl_seconds

    # ES256 JWT — header: {alg: "ES256", kid: key_id}, claims: {iss, iat, exp}
    token = sign_es256(%{"iss" => team_id, "iat" => now, "exp" => expires_at}, pem, key_id)
    {:ok, token, expires_at}
  end

  defp schedule_refresh(expires_at, existing_timer) do
    if existing_timer, do: Process.cancel_timer(existing_timer)
    ms = max((expires_at - System.system_time(:second) - @refresh_threshold) * 1_000, 0)
    Process.send_after(self(), :refresh_token, ms)
  end
end
```

**401 retry in `ExternalClient`**: On a `401` response, `ExternalClient` calls `DeveloperTokenManager.regenerate_token/0`, which immediately generates a new token, cancels the pending scheduled refresh, and returns the new token. `ExternalClient` retries the request once with the new token before returning `{:error, :unauthorized}`.

`get_token/0` (called on every normal request) does not regenerate — it only returns the cached token. Using the correct function on each path prevents unnecessary regeneration on normal requests while ensuring the 401 path always gets a fresh token.

The browser-facing developer token endpoint also calls `get_token/0` — it never generates its own token.

Required environment variables: `APPLE_MUSIC_TEAM_ID`, `APPLE_MUSIC_KEY_ID`, `APPLE_MUSIC_PRIVATE_KEY`.

### Apple Music Authentication Flow

Apple Music uses MusicKit JS (browser-initiated, v3) rather than server-side OAuth. The flow has two steps — no separate HTTP endpoint is needed for the developer token.

**Step 1 — Browser JS hook** receives the developer token as part of the LiveView event payload, runs MusicKit JS authorization, and pushes both the user token and the storefront back to the LiveView. The storefront is read directly from the MusicKit instance (`music.storefrontId`) after authorization — no separate API call is needed.

```javascript
// assets/js/hooks/apple_music_auth.js
const AppleMusicAuth = {
  mounted() {
    this.handleEvent("request_apple_music_auth", async ({ developer_token }) => {
      try {
        // MusicKit v3: configure() returns the instance directly
        const music = await MusicKit.configure({
          developerToken: developer_token,
          app: { name: "Setlistify", build: "1.0" }
        })

        const userToken = await music.authorize()
        const storefront = music.storefrontId  // available immediately after authorize()

        this.pushEvent("apple_music_authorized", { user_token: userToken, storefront })
      } catch (error) {
        // Covers user cancelling the auth dialog, MusicKit configure failures,
        // and network errors. Without this, apple_music_loading stays true forever.
        this.pushEvent("apple_music_auth_failed", { reason: error.message })
      }
    })
  }
}

export default AppleMusicAuth
```

The MusicKit v3 script tag must be added to the root layout:

```html
<script src="https://js-cdn.music.apple.com/musickit/v3/musickit.js" defer></script>
```

**Step 2 — Server LiveView handler** fetches the developer token, includes it in the push_event payload, then handles the callback with the user token and storefront:

```elixir
def handle_event("connect_apple_music", _params, socket) do
  developer_token = AppleMusic.DeveloperTokenManager.get_token()
  {:noreply,
    socket
    |> assign(apple_music_loading: true)
    |> push_event("request_apple_music_auth", %{developer_token: developer_token})}
end

def handle_event("apple_music_authorized", %{"user_token" => user_token, "storefront" => storefront}, socket) do
  user_id = UUID.uuid4()
  {:ok, user_session} = AppleMusic.API.build_user_session(user_token, storefront, user_id)
  {:ok, _pid} = AppleMusic.SessionSupervisor.start_user_token(user_id, user_session)

  socket =
    socket
    |> put_session(:user_id, user_id)
    |> put_session(:auth_provider, "apple_music")
    |> put_session(:user_token, encrypt_token(user_token))
    |> put_session(:storefront, storefront)
    |> assign(:user_session, user_session)

  {:noreply, push_navigate(socket, to: socket.assigns.redirect_to || "/")}
end

def handle_event("apple_music_auth_failed", %{"reason" => _reason}, socket) do
  {:noreply,
    socket
    |> assign(apple_music_loading: false)
    |> put_flash(:error, "Apple Music authorization failed. Please try again.")}
end
```

The storefront is stored in the session cookie so that session restoration can reconstruct the `UserSession` without an API call.

### `AppleMusic.API.build_user_session/3`

`build_user_session/3` is a pure struct constructor — it makes no network calls. Both first-auth and session-restoration call paths use it to reconstruct a `UserSession` from its component parts.

```elixir
@spec build_user_session(String.t(), String.t(), String.t()) ::
        {:ok, AppleMusic.UserSession.t()}
def build_user_session(user_token, storefront, user_id) do
  {:ok, %AppleMusic.UserSession{
    user_token: user_token,
    storefront: storefront,
    user_id: user_id
  }}
end
```

The function always returns `{:ok, session}` — there is nothing to validate at construction time. Token validity is discovered lazily on the first Apple Music API call (a `401` response triggers re-authentication). This avoids adding a network round-trip to the auth and restoration paths.

`build_user_session/3` is a callback on `AppleMusic.API` (not in `MusicService.API`, which only covers shared cross-provider operations). It is mockable via `AppleMusic.API.MockClient` in tests that exercise the auth flow.

### Sign-in UX

The sign-in page (`SetlistifyWeb.SignInLive`, routed at `/sign-in`) presents both provider options. Unauthenticated users are redirected here with a `?redirect_to=` parameter that is honoured after successful auth.

```elixir
defmodule SetlistifyWeb.SignInLive do
  use SetlistifyWeb, :live_view

  def mount(params, _session, socket) do
    {:ok,
     assign(socket,
       redirect_to: params["redirect_to"] || "/",
       apple_music_loading: false
     )}
  end

  def handle_event("connect_apple_music", _params, socket) do
    {:noreply,
     socket
     |> assign(apple_music_loading: true)
     |> push_event("request_apple_music_auth", %{})}
  end

  # apple_music_authorized and apple_music_auth_failed handlers — see Authentication Flow above

  def render(assigns) do
    ~H"""
    <div id="apple-music-auth" phx-hook="AppleMusicAuth">
      <a href="/auth/spotify">Connect with Spotify</a>

      <button phx-click="connect_apple_music" disabled={@apple_music_loading}>
        <%= if @apple_music_loading, do: "Connecting...", else: "Connect with Apple Music" %>
      </button>
    </div>
    """
  end
end
```

**Design notes**:

- The `phx-hook="AppleMusicAuth"` wrapper must have a stable DOM `id` — LiveView requires it for hook lifecycle management.
- The "Connect with Apple Music" button is a `phx-click` that sends to the LiveView, which then fires `push_event("request_apple_music_auth", ...)` to the hook. The indirection through the LiveView is intentional — it lets the server set `apple_music_loading: true` before the JS auth flow begins.
- The "Connect with Spotify" link is a standard anchor — it initiates the server-side OAuth redirect, no LiveView event needed.
- `apple_music_loading` disables the button and updates its label while waiting for MusicKit JS to complete. This prevents double-submit and gives the user feedback during the OAuth popup flow.
- On error (`apple_music_auth_failed`), `apple_music_loading` is reset to `false` so the user can retry.
- The `?redirect_to=` parameter is stored in the socket assign, not the session, to keep the session clean.
- Provider-aware display name: Spotify uses `user_session.username` in the UI; Apple Music has no username. The layout/nav component must handle `nil` or check `auth_provider` from the session rather than reading `user_session.username` directly.

### Apple Music Session Management

Apple Music user tokens are valid for ~6 months with no server-side refresh endpoint. `AppleMusic.SessionManager` is therefore a simplified GenServer compared to its Spotify counterpart — the absence of a refresh timer is the structural encoding of "no refresh needed."

```elixir
defmodule Setlistify.AppleMusic.SessionManager do
  use GenServer

  def start_link({user_id, %AppleMusic.UserSession{} = session}) do
    GenServer.start_link(__MODULE__, session, name: via_tuple(user_id))
  end

  def get_session(user_id) do
    case lookup(user_id) do
      {:ok, pid} -> {:ok, GenServer.call(pid, :get_session)}
      :error     -> {:error, :not_found}
    end
  end

  def stop(user_id) do
    case lookup(user_id) do
      {:ok, pid} -> GenServer.stop(pid)
      :error     -> {:error, :not_found}
    end
  end

  # No handle_continue(:schedule_refresh, ...) — intentionally absent
  # No handle_info(:refresh_token, ...)       — intentionally absent

  @impl true
  def init(session), do: {:ok, session}

  @impl true
  def handle_call(:get_session, _from, session), do: {:reply, session, session}

  defp via_tuple(user_id),
    do: {:via, Registry, {Setlistify.UserSessionRegistry, {:apple_music, user_id}}}

  defp lookup(user_id) do
    case Registry.lookup(Setlistify.UserSessionRegistry, {:apple_music, user_id}) do
      [{pid, _}] -> {:ok, pid}
      []         -> :error
    end
  end
end
```

On HTTP `401` from the Apple Music API, `ExternalClient` returns `{:error, :unauthorized}`. The calling LiveView handles this by clearing the session and redirecting to re-authenticate — no retry at the session layer.

### `AppleMusic.SessionSupervisor`

`AppleMusic.SessionSupervisor` mirrors `Spotify.SessionSupervisor` in structure. Both are thin wrappers around the shared `Setlistify.UserSessionSupervisor` DynamicSupervisor — they provide a provider-specific API so callers do not need to know the DynamicSupervisor's name.

```elixir
defmodule Setlistify.AppleMusic.SessionSupervisor do
  alias Setlistify.AppleMusic.SessionManager

  def start_user_token(user_id, %AppleMusic.UserSession{} = session) do
    DynamicSupervisor.start_child(
      Setlistify.UserSessionSupervisor,
      {SessionManager, {user_id, session}}
    )
  end

  def stop_user_token(user_id) do
    SessionManager.stop(user_id)
  end

  def get_session(user_id) do
    SessionManager.get_session(user_id)
  end
end
```

Both `Spotify.SessionManager` and `AppleMusic.SessionManager` run as children of the existing `Setlistify.UserSessionSupervisor` DynamicSupervisor. The `Setlistify.UserSessionRegistry` is shared. The supervisor wrapper modules exist only to provide a clean, provider-scoped API — there is no separate supervisor process per provider.

If Phase 4 review finds meaningful shared logic between the two supervisor wrappers (e.g., a `start_child/2` helper), a `Setlistify.SessionSupervisorHelpers` module can be extracted then.

### Registry Key Namespacing

Registry keys change from bare `user_id` strings to `{:spotify, user_id}` / `{:apple_music, user_id}` tuples to prevent ID collision between providers.

```elixir
# Before (Spotify only)
{:via, Registry, {Setlistify.UserSessionRegistry, user_id}}

# After (namespaced)
{:via, Registry, {Setlistify.UserSessionRegistry, {:spotify, user_id}}}
{:via, Registry, {Setlistify.UserSessionRegistry, {:apple_music, user_id}}}
```

### Session Restoration Plugs

Each provider has its own restoration plug guarded by `:auth_provider`. Both run in the `:browser` pipeline after `fetch_session`.

`RestoreSpotifyToken` gains an `:auth_provider` guard (previously unconditional):

```elixir
def call(conn, _opts) do
  if get_session(conn, :auth_provider) == "spotify" do
    # ... existing restoration logic unchanged ...
  else
    conn
  end
end
```

`RestoreAppleMusicToken` mirrors the structure. Because `storefront` and `user_id` are stored in the session cookie, session restoration requires no network call — the `UserSession` is reconstructed entirely from cookie values:

```elixir
defmodule SetlistifyWeb.Plugs.RestoreAppleMusicToken do
  import Plug.Conn
  alias Setlistify.AppleMusic.{SessionManager, SessionSupervisor, API}

  def init(opts), do: opts

  def call(conn, _opts) do
    if get_session(conn, :auth_provider) == "apple_music" do
      restore_session(conn)
    else
      conn
    end
  end

  defp restore_session(conn) do
    user_id         = get_session(conn, :user_id)
    encrypted_token = get_session(conn, :user_token)
    storefront      = get_session(conn, :storefront)

    with user_id when not is_nil(user_id) <- user_id,
         {:error, :not_found} <- SessionManager.get_session(user_id),
         {:ok, user_token} <- decrypt_token(encrypted_token),
         storefront when not is_nil(storefront) <- storefront,
         {:ok, user_session} <- API.build_user_session(user_token, storefront, user_id) do
      SessionSupervisor.start_user_token(user_id, user_session)
      conn
    else
      _ -> conn |> clear_session() |> put_flash(:error, "Session expired. Please sign in again.")
    end
  end
end
```

`build_user_session/3` is a pure constructor (no network call), so session restoration adds no latency beyond what `RestoreSpotifyToken` costs. Token expiry is discovered lazily on the first Apple Music API call.

### Telemetry Attribute Conventions

Current Spotify attributes use provider-prefixed names (`spotify.operation`, `spotify.artist`, etc.). Adding Apple Music with equivalent `apple_music.*` prefixes would create inconsistency and make cross-provider queries impossible.

**Decision**: Generalize attributes to provider-agnostic names. The span name already encodes the provider (e.g., `Setlistify.Spotify.API.search_for_track` vs `Setlistify.AppleMusic.API.search_for_track`).

| Current (Spotify) | New (Generic) | Notes |
|---|---|---|
| `spotify.operation` | *(remove)* | Redundant with span name |
| `spotify.artist` | `music.artist` | |
| `spotify.track` | `music.track` | |
| `spotify.playlist.id` | `playlist.id` | |
| `spotify.playlist.name` | `playlist.name` | |
| `spotify.playlist.url` | `playlist.url` | |
| `spotify.tracks.count` | `tracks.count` | |
| `spotify.embed.url` | `embed.url` | |
| `spotify.results.count` | `results.count` | |
| `spotify.track.uri` | `track.id` | Renamed to match new track_id convention |
| `service.name` | `service.name` | Keep — useful for span-level service filtering |

SetlistFM attributes (`setlist_fm.operation`, `setlist_fm.search.query`, etc.) are already service-prefixed for a service where there is no cross-provider concern. These are left unchanged.

The `show_live.ex` async song-search span gains `{"music.service", service_name}` where `service_name` is derived from the user session type. This enables filtering all song-search spans by provider in Grafana without having to join child spans.

## Development Guidelines

These are standing rules for all implementation work in this project, not specific to this ADR.

### Test-Driven Development

Write a failing test before writing any implementation code. No exceptions. The test defines the contract; the implementation satisfies it.

- For new modules, write the test file first and let compile errors and test failures drive the interface design
- For refactors (e.g., renaming `:uri` → `:track_id`), update the test expectations first, confirm they fail, then update the implementation
- `render_async/1` and `set_mox_from_context` are already in place for async LiveView tests — use them

### Commit and PR Granularity

Be intentional about decomposing work into small, independently-reviewable commits and PRs. If a change stands alone — it compiles, tests pass, and it makes a coherent unit of work — it should be its own PR.

Examples of changes that warrant their own PR:
- Deleting the `MusicAccount` stub
- Removing `preview_url` from the search result type
- Renaming `:uri` → `:track_id`
- Adding `:auth_provider` to the Spotify session
- Namespacing Registry keys

Phases are a useful grouping, but a phase is not the unit of a PR. Prefer multiple small PRs per phase over one large PR per phase.

### Pre-Commit Checklist

Run all of the following before every commit:

```shell
mix format          # fix formatting — do not commit unformatted code
mix dialyzer        # no new warnings
mix test            # all tests green
```

If any step fails, fix it before committing. Do not use `--no-verify` or skip steps. If `dialyzer` reveals a type issue introduced by the change, fix the type issue rather than suppressing the warning.

## Future Considerations

1. **Additional providers** — The dispatch pattern is open for extension (e.g., Tidal, YouTube Music): add a `UserSession` struct, implement `@behaviour MusicService.API`, add two dispatch clauses
2. **Apple Music re-auth UX** — Currently a 401 forces full re-auth. A future improvement could check token age against the session cookie's issue timestamp and prompt re-auth proactively before the first API failure
3. **Storefront defaulting** — If the storefront value is missing from the session cookie during restoration, defaulting to `"us"` would prevent an unnecessary sign-out for an edge case that shouldn't occur in practice
